环境：WSL2 / Ubuntu，Node.js，Claude CLI

## 测试案例 1：Landing Page（简单任务）

**输入任务：**
> please create a landing page for Fractals

**配置：** maxDepth=3，Executor=Claude

**结果：** 失败

GPT 将任务分类为 `atomic`，整个任务作为单个叶子交给 Claude 执行。Claude 退出码为 0，但 output 仅 429 字节（纯文字描述），worktree 中无任何文件产出。

**根本原因：** 任务描述过于简单，GPT 未触发拆解；Claude 在 `-p` 非交互模式下对模糊任务只输出文字，不执行写文件操作。

## 问题与改进建议

### 问题 1：任务粒度敏感
- **现象：** 任务描述过于宽泛时，GPT 分类为 `atomic`，Claude 不写文件
- **建议：** 用户需要明确列出需要哪些文件/模块，才能触发有效拆解；或在 UI 中提示用户任务描述规范

### 问题 2：子任务产出需手动合并
- **现象：** 每个子任务产出在独立 worktree（`.worktrees/1.1`、`.worktrees/1.2`...），无自动合并
- **影响：** 用户需手动收集各 worktree 的文件才能得到完整项目
- **建议：** 实现 roadmap 中的"Backpropagation merge agent"

### 问题 3（UI）：Confirm & Execute 按钮位置不显眼
- **现象：** 任务分解完成后，"Confirm & Execute" 按钮位置靠下，不熟悉的用户容易误以为任务已开始执行而干等
- **建议：** 将确认按钮置顶或添加明显的视觉提示，告知用户"分解完成，请确认后执行"

## 总结

| 维度 | 评价 |
|------|------|
| 核心机制 | 有效，递归拆解 + 并行 worktree 执行思路成立 |
| 代码质量 | 子任务产出的代码可直接运行，质量较高 |
| 易用性 | 需要用户了解任务描述技巧，学习曲线存在 |
| 成熟度 | 实验性项目，缺少自动合并、持久化等关键特性 |
| 适用场景 | 适合结构清晰、模块独立的任务；不适合高度耦合的单体任务 |
